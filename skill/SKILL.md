# 微信公众号技术内容创作助手

## 概述

帮助用户创作高质量的技术类公众号文章。支持三种风格，根据内容自动匹配。

## 自定义系列（可选）

如果你有一个固定的内容系列，可以在此定义系列规则。当用户提到系列名、相关关键词，或从选题表中选题时，默认进入本系列规则。

按以下模板填写你自己的系列信息：

```yaml
# ----- 系列配置（请替换为你自己的） -----
系列名: "你的系列名"                          # 例如："普通人的 AI 实战"
副标题: "你的系列副标题"                      # 例如："从真实场景出发，教你把 AI 用起来。"
核心读者: "你的目标读者群体"                  # 例如："AI 小白、普通上班族、学生"
内容目标: "你的内容目标描述"                  # 例如："不讲概念，从场景出发，教读者把 AI 用起来"
写作原则: "你的写作原则"                      # 例如："场景是入口，工具是手段，步骤是交付，认知是复利"
```

> 如果你不需要系列规则，可以删除本节。系列规则仅在你希望统一多篇文章的标题格式、读者定位和写作调性时使用。

### 场景化前置分析（写作前必须完成）

正式写正文前，先用简短小节明确：

1. 读者是谁：这篇主要写给哪类普通人。
2. 真实场景：他在什么情况下遇到这个问题。
3. 常见卡点：以前为什么做不好、做不动、做不清楚。
4. AI 帮哪一步：AI 负责整理、生成、改写、陪练、分析、检查，还是流程串联。
5. 最终交付物：读者最后能拿到什么，例如清单、表格、提示词、模板、文档、脚本。
6. 可复制步骤/提示词：文章必须给到能照着做的步骤或提示词，不做纯观点文。

### 隐形文章骨架

文章可以写成故事型、教程型、测评型、拆解型、问答型或复盘型，但背后必须覆盖这些要素，不要机械露出模板痕迹：

`场景进入` → `卡点还原` → `AI介入` → `实操演示` → `结果交付` → `避坑提醒` → `评论区征集下一个场景`

## 第一步：风格识别与素材分析

### 输入处理
- 网站链接 → 使用 web_fetch 获取完整内容
- 图像素材 → 识别内容提取关键信息
- 文本素材 → 先判断类型（见下方），再处理

### 输入类型判断 ⚠️ **关键步骤**

| 输入类型 | 识别信号 | 处理方式 |
|---------|---------|---------|
| **零散素材** | 关键词、片段想法、一句话灵感、要点列表 | 提炼核心，从零创作 |
| **结构化素材** | 有段落但不完整、笔记形式、提纲 | 补充扩展，按风格结构化 |
| **现有文章** | 完整的开头结尾、段落结构清晰、已经像一篇成品 | **重写而非重排**（见下方规则） |
| **无法判断** | — | 询问："这是一篇已有文章需要改写，还是素材需要创作？" |

**⚠️ 现有文章重写规则（必须遵守）**：

当输入是一篇已有的完整文章时，必须按以下方式处理，**不是重新排版**：

1. **提取内核** — 这篇文章的核心观点/信息/数据是什么？列出来
2. **打碎重组** — 丢掉原文的结构和表达方式，用 B/D/E 对应的风格结构重新组织
3. **换人设说话** — 原文是别人写的，重写后必须是"我"在说，用口播DNA的语气
4. **加入个人视角** — 至少加入2-3处原文没有的个人判断/感受/踩坑经历
5. **补充新信息** — 通过搜索补充原文没有的最新数据/案例/工具版本
6. **去重检测** — 重写后的句子不能和原文超过连续15字相同

**简单判断标准**：如果输出和输入的段落结构基本一致，说明你只是在重排，不是重写。

### 快速需求确认

仅在信息严重不足时询问（不超过2个问题）：
- 目标读者？（普通用户 / 技术开发者 / 行业人士）
- 文章长度？（短文800-1500字 / 中文1500-3000字 / 长文3000+字）

信息充足时直接开始创作。

## 第二步：内容研究

⚠️ **无论输入是原始素材还是现有文章，都必须执行内容研究**

**检查清单**：
- [ ] 是否使用了 WebSearch 搜索最新行业动态？
- [ ] 是否补充了时效性信息（数据/案例）？
- [ ] 是否验证了文章中提到的工具/技术的最新版本？

**如果跳过此步骤，需要在输出中说明理由**

**搜索策略：**

- 搜索技术文档、GitHub仓库、开发者博客、官方changelog
- 搜索技术社区讨论（知乎、V2EX）
- 查找竞品/同类工具的对比信息

## 第三步：按风格创作

⚠️ **开始写作前必须执行以下两步，不能跳过**：

1. **读取** `references/writing-style.md` — 获取对应风格的完整定义：语气基调、开头公式、写作铁律、禁忌清单、自检清单
2. **读取** `references/books.md` — 根据文章风格选取合适的引用观点和金句

> 这两个文件是写作的核心规范。直接从记忆写作会丢失大量细节，尤其是句式禁忌和开头公式。

⚠️ **金句引用不重复规则（必须遵守）**：
- 每篇文章引用的书籍金句必须**轮换使用**，同一句金句不得在连续文章中重复出现
- 同一本书的不同金句可以用，但同一句话不能反复用
- books.md 每本书提供了多条金句，每次写作时主动选择**未用过的**金句
- 如果拿不准之前用过哪些，优先选该书中排序靠后的、不太常见的金句

### 🧑‍💻 风格B：个人技术博客

**文章结构**：
1. Hook开头（50-100字）→ 从 `writing-style.md` 中选一个B类开头公式
2. 背景铺垫（100-200字）→ 为什么写这篇、事情来龙去脉
3. 核心内容（主体）→ 技术细节+实操过程+截图/代码，小标题分段
4. 个人观点/总结（100-200字）→ 我的看法、适合什么人用、下一步怎么做
5. 互动引导 → "你用过吗？评论区聊聊"

**📢 口播DNA规则（此处独有，writing-style.md 没有，必须遵守）**：

1. **人称体系**：用「咱」替代「我们」，「你」字高密度出现，每2-3段至少一次
2. **反问推进**：核心观点后用反问收尾——"你说这事儿你能接受吗？""这放到你身上你怎么办？"（每3-4段一次）
3. **数字具象化**：❌ "大幅提升效率" → ✅ "以前手动整理要2小时，现在15分钟"
4. **自我暴露**：每篇至少一处承认局限——"说实话这个坑我踩了两次""这功能我现在也没完全摸透"，暴露后紧接解决方案
5. **历史类比**：引入新工具时找读者已有的参照系——"这就跟Excel的VLOOKUP差不多，只不过它可以理解自然语言"
6. **情绪安抚**：遇到复杂内容主动降温——"看着步骤多，实际跑一遍10分钟""先跑起来再说，不用全部搞懂"

> 写作铁律、反面教材、句式禁忌详见 `references/writing-style.md` B 类

### 💡 风格E：观点洞察

**文章结构**：
1. 事件锚点（100-150字）→ 从 `writing-style.md` 中选一个E类开头公式，具体到人名/公司/数字
2. 问题抛出（50-80字）→ 点出核心矛盾，为什么值得讨论
3. 多角度分析（400-600字）→ 2-3个维度，每个维度至少一个真实案例
4. 我的判断（100-150字）→ 明确表态，说清楚为什么这么想，允许"我不确定"
5. 给读者一句话（50-80字）→ 可操作的思考方向，不喊口号

## 第四步：标题创作

### 系列标题（可定制）

如果你配置了自定义系列，默认优先使用系列标题格式，同时保留下方备选标题机制。

**系列标题格式**（请替换为你的系列名）：

```text
{你的系列名} XX：{根据文章内容制定的自然标题}
```

**执行规则**：
- 系列名：替换为你自己的系列名（如不需要系列，可删除本节）
- 编号默认使用两位数：`01`、`02`、`03`
- 冒号使用中文冒号：`：`
- 后半标题必须根据文章内容单独制定，落到具体读者、具体场景、具体问题或具体结果
- 不固定使用"怎么用 AI 解决/完成 XXX"，避免模板腔
- 如果用户提供系列编号，优先使用用户编号
- 如果用户没有提供编号，根据文章保存目录、上下文或选题表推断；无法推断时先用 `XX` 占位并在交付中说明
- 原有标题格式继续保留，作为传播标题、SEO 标题、备选标题使用

**示例**（请替换为你自己的系列名和内容）：

```text
{你的系列名} 01：不会写周报？让 AI 先帮你把流水账变成成果
{你的系列名} 02：孩子背古诗总忘？用 AI 做一个陪练小老师
{你的系列名} 03：资料太多看不完？让 AI 先帮你拆成重点清单
```

## 第五步：排版与交付

**排版建议**（所有风格通用）：
- 短段落：手机端每段不超过3-4行
- 小标题：每500字左右一个小标题，可用emoji增强识别度
- 重点加粗：核心观点、关键数据
- 配图节奏：每500-800字配1张图（位置建议）
- 代码块：技术文章的代码用等宽字体展示

**固定模板（请替换为你的账号名）**：

```
大家好，我是{你的账号名}。

[文章正文]

---
**关注公众号「{你的账号名}」**，和 AI 一起成长。
有问题或建议？后台留言即可。
```

**最终交付物**：
1. 完整文章正文（标题即为文件名，首行不再出现标题）
2. 3个备选标题方案
3. 摘要（100字以内，用于分享时的描述）
4. 配图建议（详见 `references/image-guide.md`）
5. 评论区互动话题建议

**输出格式**：
- 短文（<1500字）→ 直接在对话中输出
- 长文（>1500字）→ 保存到 `{vault根目录}/01_内容创作/02_内容生产中/YYYYMMDD-标题/01_文章内容/公众号文章.md`，同时生成 `01_文章内容/配图建议.md`

**文件命名规范**：
- 日期格式：YYYYMMDD（如20260327）
- 标题格式：使用英文连字符，避免特殊符号
- 示例：`20260327-镜头语言让AI视频有电影感/01_文章内容/公众号文章.md`
- ❌ 错误示例：`20260327-镜头语言让AI视频有电影感.md`（没有单篇内容结构）

**文件夹结构**（每篇文章必须创建）：
```

├── 00_内容卡片.md
├── 01_文章内容/
```

⚠️ **必须创建单篇内容结构**，不要直接保存到内容生产中的根目录，也不要写入旧目录

**内容卡片与文章 frontmatter 同步规则（必须遵守）**：

内容卡片（00_内容卡片.md）是元数据中枢，必须包含 `title` 和 `description` 字段：

```yaml
---

description: 公众号摘要，100字以内，用于分享时的描述
series: {你的系列名}

...

```

公众号文章（公众号文章.md）必须有 frontmatter，包含 `title` 和 `description`，从内容卡片同步：


⚠️ **配图自动生成流程**：

**默认行为：自动生成配图，无需用户选择风格**

默认使用「简笔流程」风格（合并自单色流程 + 简约手绘），文章创作完成后自动进入配图生成环节，不需要询问用户。

**Step 1：分析文章结构，生成配图**

读取 `references/image-guide.md`，按以下流程自动执行：

1. 按文章 H2/H3 标题拆分段落，提取核心观点和关键概念
2. 根据段落内容匹配图表类型（概念图/流程图/对比图/层次图/关系图）
3. 使用「简笔流程」风格的全局风格指令 + 对应图表类型的提示词模板，生成完整中文提示词
4. 展示配图建议大纲，等待用户确认
5. 确认后生成 `配图建议.md` 保存到文章目录

⚠️ **封面图必须包含文章标题（必须遵守）**：
- 封面图的提示词中**必须明确要求将文章标题文字融入画面构图**
- 标题文字应作为封面的**视觉主体/焦点**，用较大的手写风格字体呈现
